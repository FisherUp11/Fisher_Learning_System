# 04｜内容导入与实施路线图

## 1. 先用 30 个字试跑，不要直接导入 1300 个

第一批建议选择孩子常见、可组成短词的 30 个字，例如：人、口、手、日、月、山、水、火、木、土、大、小、上、下、中、天、地、我、你、他、爸、妈、家、爱、看、听、说、来、去、好。

这 30 个字能暴露真正的问题：字形是否清晰、拼音是否正确、例词是否适龄、朗读是否自然、一次学习多少会疲劳、家长是否看得懂“会/不会”。试跑 7 天后再导入 1300 字，返工成本最低。

## 2. 汉字导入 CSV 规范

### 最小可用字段

```csv
character,pinyin_marked,meaning,package_code,sequence,level
人,rén,人；人类,preschool-1300,1,学前
口,kǒu,嘴；开口说话,preschool-1300,2,学前
```

| 字段 | 必填 | 规则 | 示例 |
| --- | --- | --- | --- |
| `character` | 是 | 单个汉字；去空格；全库唯一 | `山` |
| `pinyin_marked` | 是 | 带声调的完整拼音；多音字用首选读音 | `shān` |
| `meaning` | 是 | 儿童能理解的短释义，建议 ≤ 24 字 | `高高的山` |
| `package_code` | 是 | 内容包稳定编码 | `preschool-1300` |
| `sequence` | 是 | 正整数，决定默认学习顺序 | `18` |
| `level` | 否 | 学前/一年级等 | `学前` |

### 推荐增强字段

```csv
character,pinyin_marked,pinyin_numbered,meaning,radical,strokes,word_1,word_2,example_sentence,package_code,sequence,level,notes
山,shān,shan1,高高的山,山,3,大山,山上,山上有一棵树。,preschool-1300,18,学前,
```

- `pinyin_numbered`：供程序解析声调与检索，例如 `shan1`。
- `radical`、`strokes`：先只显示，笔顺教学第二阶段再用。
- `word_1`、`word_2`：优先人工提供儿童常用词；AI 可补候选但不覆盖人工内容。
- `example_sentence`：一句短句，尽量只使用已学/容易理解的字。
- `notes`：内容管理员备注，不在孩子端显示。

### 多音字处理

一个 `characters` 主记录只表示字形；读音做子记录或在扩展表保存。例如“行”有 `xíng` / `háng`，内容包中应指定当前学习的首选读音与例词。不要用一个拼音字段硬塞所有读音并让儿童卡片同时展示。

## 3. 导入流程

```mermaid
flowchart LR
  A[上传 CSV] --> B[格式校验]
  B --> C[预览：新增/更新/错误]
  C --> D[导入为草稿包]
  D --> E[抽样检查：拼音、例词、朗读]
  E --> F[发布给指定孩子]
  F --> G[开始排入每日新字]
```

### 校验项

- UTF-8 编码、表头正确、文件大小和行数在上限内。
- `character` 恰好一个汉字；同一个文件内不重复。
- 拼音有声调或能从数字声调标准化；不能解析的行标红。
- `sequence` 不重复，或明确提示系统将按导入顺序补齐。
- 更新已发布字时显示字段差异；不能悄悄改变已经学习过孩子的标准拼音。
- 导入失败必须是“整批不发布”，错误行可导出；不要半成功半失败。

## 4. AI 内容的批量生成与审核

建议每个字只生成以下候选，避免生成过多内容把卡片变复杂：

```json
{
  "example_sentence": "山上有一棵树。",
  "association_tip": "山像高高的三座小山峰。",
  "poetry_hint": {
    "text": "白日依山尽",
    "title": "登鹳雀楼",
    "author": "王之涣"
  }
}
```

审核清单：

- 句子是否适合孩子年龄，是否短且自然。
- 是否把汉字读音、字义、古诗出处说错。
- 是否有危险、恐惧、成人、歧视、商业或不合适的内容。
- 古诗句是否只作为“关联提示”，不把单个字的解释牵强附会。
- 一次发布前抽查至少 10% 的生成内容；对有疑问的字采用人工内容或不显示 AI 卡。

## 5. 音频内容规范

### 朗读音频

- 单字：标准普通话，单声道即可，保留适量前后静音。
- 例词/例句：只为孩子真正会点开的内容生成；缓存避免重复付费。
- 资源命名不含孩子昵称，例如 `characters/{character-id}/single.mp3`。

### 家长导入音乐（P2）

推荐字段：`title, performer, source_note, lyrics_or_text, cover_image, mp3`。

- 只上传家庭有权使用的内容。
- MP3 上传到私有 bucket，播放时服务端签发短时 URL。
- 若有歌词或诗歌文本，先人工分段；不要用音频自动转写作为第一版依赖。
- 文件大小/时长设上限；导入后转码、波形图、离线下载都可后置。

## 6. 实施路线图与验收

### Phase 0：内容与交互原型

**范围**：30 字 CSV、静态儿童首页、两种卡片状态、家长概览静态数据。

**验收**：在真实 iPhone 与 iPad 上，孩子不用解释即可完成“听、看、我认识/再学一次”；家长能一句话说出明天该做什么。

### Phase 1：汉字 MVP

**范围**：登录、孩子档案、内容包、导入预览、今日队列、服务端调度、尝试记录、朗读、基础进度、PWA 安装。

**验收清单**：

- [ ] 1300 字可导入为草稿并发布，不产生重复字或错误顺序。
- [ ] 新字和到期复习按设定上限组成今日队列。
- [ ] 关闭/刷新后，已答题不丢失、未答题不误记。
- [ ] “再学一次”会缩短下次间隔且留下记录；“我认识”会推进阶段。
- [ ] 家长能找到某个字的全部学习尝试与下次日期。
- [ ] AI / Speech 不可用时，字卡、答案、自评和保存仍可完成。
- [ ] 私有媒体不可通过公开 URL 访问；家长 A 不能看到家长 B 的孩子数据。
- [ ] 用两个真实测试账号验证：跨家庭的内容草稿、学习记录、会话和媒体读写均被 RLS 拒绝。

### Phase 1.1：提高内容质量

**范围**：拼音拆解、家长审核 AI 内容、薄弱字建议、数据导出、可选应用内提醒。

**验收**：一位家长可以无需数据库工具完成“导入 → 审核 → 发布 → 看建议”的全流程。

### Phase 2：古诗

**范围**：诗词导入、诗词学习卡、分句朗读、自评背诵、同一复习队列。

**验收**：同一孩子的汉字与诗词到期任务合并排序；家长能分别查看两类熟练度。

### Phase 3：音乐

**范围**：私有 MP3 上传与播放、歌词/诗歌文本、会唱/会背自评、复习提醒。

**验收**：音频归属、删除、签名播放链接和学习记录在 iPhone/iPad 上均正确。

## 7. 作为产品负责人需要养成的判断习惯

1. **每个新功能先写“它改变哪一个孩子动作？”** 说不清就暂缓。
2. **把“完成”与“认识”分开。** 学过一张卡不代表记住；数据库必须记录回答，不只记录进度百分比。
3. **先做可解释的算法。** 家长能理解“明天/三天后再见”，比一开始用黑箱模型更有价值。
4. **内容质量先于 AI 数量。** 1300 个可信的拼音和例词，胜过 1300 段未经审核的生成内容。
5. **在真实设备上验收。** iPad 浏览器里的好看布局，未必在 iPhone 的安全区、键盘、网络和音频规则下好用。
6. **把家长的设置变成默认值。** 默认 5 个新字、15 个复习、一次只做一个选择；高级调节藏在家长模式。
